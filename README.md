# An Equilibrium Solver for Dynamic Queueing Games


The software and data in this repository are a snapshot of the software and data that were used in the research reported in the paper [An Equilibrium Solver for a Dynamic Queueing Game](https://dx.doi.org/10.2139/ssrn.5282750) by Denise Cerna, Chiwei Yan, and Hongyao Ma.

## Cite

To cite the contents of this repository, please cite both the paper and this repo, using their respective DOIs.

https://dx.doi.org/10.2139/ssrn.5282750

## Description

There are two goals to this repository. The first is to replicate the numerical experiments in the paper "An Equilibrium Solver for Dynamic Queueing Games" by Denise Cerna, Chiwei Yan, and Hongyao Ma. The second is to provide a tool that analyzes equilibrium behavior in queueing systems with strategic agents through analytical policy iteration methods. The implementation supports multiple dispatching rules with applications to real-world data from Chicago ride-sharing operations.

## Overview

This repository contains the implementation and computational experiments for the analytical policy iteration framework described in our paper. The code supports the analysis of strategic queueing systems where agents make optimal decisions regarding item acceptance and queue joining behavior under various dispatching policies.

## Replicating results

The scripts are located in the `scripts` folder, the source code for the algorithms and methods developed in the paper is located in the `src` folder, and the data is located in the `data` folder. All result files are outputed in a `*.pkl` format in the corresponding `completed_runs` folder. The final plots generated by the scripts are saved in the corresponding `figures` folder:

- To replicate the results in Section 5.1 (Varying Driver Supply), run the scripts in the `experiments_5_1` folder. Then, run `plot_5_1.py` to create the plots.
- To replicate the results in Section 5.2 (Mis-specified Dispatch Policies), run the script `run_mis_specified.py`.  Then, run `plot_5_2.py` to create the plots.
- To replicate the results in Section 6.1 (Parallel Dispatching), run the script `run_competition.py`.  Then, run `plot_6_1.py` to create the plots.
- To replicate the results in Appendix D.2, run the script `run_front_of_queue_joining.py`.  Then, run `plot_D_2.py` to create the plots.


**Note**: It may take hours to replicate some of the results in the paper. Notably, the Pure Random dispatch policy in the Section 5.1 results takes a long time to converge depending on the initializaiton. Furthermore, the largest memory requirement occurs in the replication of Section 5.2, which requires the analysis of a queue length (Qmax) up to 1,200. The gradient algorithm was run using 16 threads to improve speed of computation. To quickly test the scripts, change the simulation parameters to relax the convergence criteria or the maximum number of runs.  

To run the entire process outlined below automatically, run the script `run_all.py`.
 

## Repository Structure

### Source Code (`src/`)

#### `simulator_class.py`
Main simulation engine implementing four solution algorithms:
- **0-MIX**: Approximate Projection Algorithm (Algorithm 2)
- **0-GRAD**: Gradient Projection Algorithm (Algorithm 1)
- **0-MIX-GRAD**: Gradient Projection Algorithm with warm start from Approximate Projection Algorithm 
- **0-COMP**: Competition experiements

#### `setting_class.py`
Configuration management for experimental parameters:
- Economy parameters
- Qmax (An arbitrary upper bound on the possible length of the queue)
- Dispatching rule specification
- Step size control
- Initialization strategies: uniform acceptance ('1') or random ('RAND')

#### `dispatching_class.py`
Dispatching policy implementation supporting:
- `STRICT_FIFO`: Standard first-in-first-out
- `LIFO`: Last-in-first-out
- `DIRECT_FIFO`: Direct FIFO
- `PURE_RAND`: Pure random 
- `DYNAMIC_RAND_FIFO`: Dynamic randomized FIFO
- `COMP`: Competition (Must specify two dispatching policies)

#### `system_class.py`
Core system solver implementing:
- Rate calculations for state transitions
- Linear system formulation (Ax = b)
- Recursion for continuation payoff calculation

#### `outcome_class.py`
Results container storing:
- Final Value function (V), Acceptance policy, Joining decision
- Performance metrics

#### `reporting_class.py`
Analysis and visualization tools for:
- Data frame generation
- Results visualization

#### `economy_class.py`
Economy parameter management and market structure definition.

#### Utility Functions

- `get_economies.py`: Chicago ridesharing data parameter extraction
- `continue_run.py`: Checkpoint and restart functionality
- `utilities.py`: Additional helper functions

#### Input Data (`data/`)
- `trips_from_OHare_by_dropoff_community_area_for_Python.csv`

Chicago ride-sharing trip data used for parameter calibration and real-world validation.

#### Plot a Run
Generated continuation payoff, acceptance and joining decisions, and performance metrics for an individual run that is being plotted using `plot_a_run_ipynb`.


### Core Experiments (`scripts/experiments_*/run_*.py`)
Comprehensive analysis of different dispatching policies using O'Hare airport data:
- `run_strict_fifo.py`: Strict FIFO dispatching
- `run_lifo.py`: LIFO dispatching
- `run_direct_fifo.py`: Direct FIFO dispatching
- `run_dynamic_rand_fifo.py`: Dynamic Randomized FIFO dispatching
- `run_pure_rand.py`: Pure random dispatching
- `run_mis_specified.py`: Misspecified example
- `run_competition.py`: Parallel dispatch
- `run_front_of_queue_joining.py`: Front-of-Queue joining

### Figure Generation (`scripts/create_plots/plot_*.py`)
Results analysis and figure generation. Must be run after the scripts in the `scripts/experiments_*` folders.

## Key Decision Variables

### Strategy Variables
- **alpha**: Acceptance probability mapping from item types to queue states (q, Q)
- **phi**: Joining strategies indexed by queue position [0...Qmax]

### Performance Metrics
The framework computes equilibrium values for:
- Queue length 
- System throughput
- Net Revenue 
- Average waiting time

## Usage Instructions

### Experimental Configuration
Experiments typically use parameters from Chicago ride-sharing data:

```python
# Load calibrated economic parameters
economy = get_ohare_economy()
earnings = economy.w.tolist()
job_rates = economy.mu_jobs.tolist()
waiting_cost = economy.c
patience = economy.patience

# Set experimental parameters
dispatching_rule = "PURE_RAND"  # or other dispatch rules
tolerance = 1e-4
max_iterations = 2000
```

### Running Experiments
Standard workflow:

```python
for driver_arrival_rate in range(1, 16):
    Qmax = ohare_get_Qmax(driver_arrival_rate, dispatching_rule)
    
    setting = Setting(earnings, job_rates, driver_arrival_rate, 
                     waiting_cost, patience, Qmax, dispatching_rule,
                     initial_alpha='RAND')
    
    system = System(setting)
    simulator = Simulator(system, max_iterations, tolerance, 
                         '0-MIX-GRAD', saving_multiple)
    
    outcome_list, V, alpha, phi, num_iter, _ = simulator.run_iterations()
    filename = save_run_history(simulator.setting, outcome_list, 
                               simulator.version, setting.dispatching_rule)
```

### Algorithm Parameter Tuning
The framework supports extensive algorithmic parameter customization:

```python
setting.beta = 0.9                    # Momentum parameter
setting.gamma_cnst = 20               # Constant step size
setting.gamma_exp = 0                 # Step size exponent (0 = constant)
setting.gamma_phi_divide_by = 20      # Joining strategy step size
setting.momentum_start_iter = 20      # Iteration to begin momentum
setting.num_parallel_workers = 10     # Parallel processing threads (useful for gradient algorithm)
setting.gamma_cnst_grad = 200         # Gradient algorithm step size
```

### Results Analysis (`scripts/plot_a_run.ipynb`)
Generate visualizations of a specific run:

```python
reporting = Reporting(filepath)
reporting.plot_run()
```

### Convergence Parameters
- Default termination threshold: 10^⁻4
- Recommended step size: Details are listed in Appendix D.3 of the paper.
- Momentum parameter β: Problem-dependent tuning required

### Output Archive
- Default output: `intermediate_saved_results/` folder
- Archive fully converged runs in `completed_runs/` folder

### Checkpointing
The framework supports experiment interruption and restart through the checkpoint system implemented in `continue_run.py`.
